3. NOTE SEARCH ALGORITHM COMPARISON (Execution Timeline)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

SCENARIO: Song with 10,000 notes, input at position 5,000 (middle)
Search window: ±2 seconds (usually ~100-200 notes within window at typical BPM)

LINEAR SCAN (O(n)):
┌──────────────────────────────────────────────────────────────────┐
│ Comparisons needed: ~5,000 average (half of total)               │
│ Time: HIGH (slower games or mobile)                              │
│ ↓                                                                 │
│ [====== Check position 0 ======]                                │
│ [====== Check position 1 ======]                                │
│ ...                                                              │
│ [====== Check position 4999 ======] ← Finally in range           │
│ [====== Check position 5000 ======] ✓ FOUND                      │
│ [====== Check position 5001 ======] ✓ FOUND                      │
│ ...                                                              │
│ [====== Check position 5199 ======] ✓ FOUND                      │
│ [====== Check position 5200 ======] ✗ Out of range, continue    │
│ [====== Check position 10000 ======]                             │
└──────────────────────────────────────────────────────────────────┘

BINARY SEARCH (O(log n)):
┌──────────────────────────────────────────────────────────────────┐
│ Comparisons needed: ~14 to find position, then expand            │
│ Time: GOOD (most games)                                          │
│ ↓                                                                 │
│ Step 1: Check position 5000 (middle)      ✓ In range             │
│ Step 2: Check position 2500 (below range) ✗ Too early            │
│ Step 3: Check position 7500 (above range) ✗ Too late             │
│ ... (continue binary search for ~14 steps total)                 │
│ Found insertion point at position ~5000                          │
│                                                                  │
│ Then expand window:                                             │
│ Check 4900-5100 range: ~100 notes found ✓                       │
└──────────────────────────────────────────────────────────────────┘

SPATIAL BUCKETING (O(1) average):
┌──────────────────────────────────────────────────────────────────┐
│ Bucket size: 0.1 seconds (time-based)                            │
│ Song duration: 300 seconds = 3000 buckets                        │
│ ↓                                                                 │
│ Input at time 234.56 seconds:                                   │
│ Bucket ID = floor(234.56 / 0.1) = 2345                          │
│ Search buckets: 2343-2347 (±2 buckets for ±0.2s window)         │
│ Total comparisons: ~5-20 notes (direct access to buckets)        │
│ Time: EXCELLENT (modern games, streaming scenarios)              │
└──────────────────────────────────────────────────────────────────┘

6. SIMULTANEOUS INPUT HANDLING
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Scenario: Player presses 3 buttons at the same frame, 3 notes nearby

Frame time: 2.000s

Available Notes:
  Note A: 1.990s (delta: -10ms)  ← Closest
  Note B: 2.005s (delta: +5ms)   ← Second closest
  Note C: 2.050s (delta: +50ms)  ← Third closest

User Inputs:
  Input 1: Lane 0, time 2.000s
  Input 2: Lane 1, time 2.000s
  Input 3: Lane 2, time 2.000s

Matching Algorithm:

  FOR each input (sorted by delta to nearest note):
    
    Input 1 (Lane 0):
      Best match: Note A (delta -10ms = PERFECT)
      Result: HIT! Mark Note A as matched
      │
      ├─ Note A removed from available pool
      │
    Input 2 (Lane 1):
      Candidates: Note B, Note C (Note A now matched)
      Best match: Note B (delta +5ms = PERFECT)
      Result: HIT! Mark Note B as matched
      │
      ├─ Note B removed from available pool
      │
    Input 3 (Lane 2):
      Candidates: Note C only (A & B matched)
      Best match: Note C (delta +50ms = GOOD)
      Result: HIT! Mark Note C as matched
      │
      └─ All inputs consumed, all notes matched!

7. LATENCY COMPENSATION PROCESS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

User presses button at REAL time t = 0.000s
But the game detects it later after accounting for various delays:

REAL WORLD TIMELINE:
═══════════════════════════════════════════════════════════════════

t=0ms:    ▓ User presses button
          
          ~5-20ms: Button signal travels through hardware/USB
          ▓ Input device detects press
          
          ~10-50ms: Audio latency (system buffer)
          ▓ Audio actually plays through speakers
          
          ~16-100ms: Display lag (GPU, monitor refresh)
          ▓ Visual feedback displayed on screen
          
          ~30-170ms: TOTAL LATENCY

═══════════════════════════════════════════════════════════════════

COMPENSATION IN GAME:

Without compensation:
  Real press time: 0.000s
  Audio playback time: +50ms (buffered)
  Game sees press at: 0.050s
  Result: Registers 50ms LATE on target notes (inaccurate!)

With compensation (CORRECT):
  Raw input time: 0.050s (as detected by game)
  Audio latency offset: -0.050s (compensate backwards)
  Adjusted time: 0.050 - 0.050 = 0.000s
  Result: Correctly aligned with music (accurate!)

CALIBRATION PROCESS:

User enters calibration mode:
  1. Visible click appears on screen
  2. User presses when they hear the sound
  3. Game measures: audio_time - input_time = offset
  4. Offset stored and applied to all future inputs
  
Example:
  User presses at t=2.500s (as detected)
  Audio marker was at t=2.450s
  Offset = 2.500 - 2.450 = +50ms
  Store: latency_offset = 0.050
  
Future calculations:
  input_time - latency_offset = adjusted_time
  2.500 - 0.050 = 2.450 ✓ Correct!
